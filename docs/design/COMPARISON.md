# Lynceus UI — Design Reference & Parity Index

This directory holds the **UI design source of truth** for Lynceus, imported from the
claude.ai/design handoff (`design_handoff_lynceus_monitoring`) on **2026-07-10**.

Its purpose is durable, version-controlled reference so **any session can compare the
current frontend against the intended design** without re-importing the bundle.

## Contents & read order

1. **`PRODUCT_INTENT.md`** — the *why*: decision log from the design conversation.
   **Source of truth for intent.** Where the prototype and this doc disagree, this wins;
   flag the discrepancy.
2. **`README.md`** — the *what*: handoff overview, screen-by-screen spec, scope model,
   design tokens, tweakable props.
3. **`Lynceus.dc.html`** — the high-fidelity interactive prototype (template + logic +
   demo data). The pixel/token/interaction truth. Open in a browser to compare.
4. **`support.js`** — prototype runtime. **Reference only**, not code to port.

Read order for a fresh session: `PRODUCT_INTENT.md` → `README.md` → open `Lynceus.dc.html`.

## Ground rules (from the handoff)

- These are **design references**, not production code. Recreate faithfully in the
  codebase's stack (**templ + HTMX**, see `web/`), not by copying the HTML/JS.
- **Fidelity is high** — colors, typography, spacing, interaction patterns are final.
- All demo data in the prototype is hardcoded; replace with real APIs.
- Privacy backbone still governs: only T1 (normalized, literal-free) data renders unless
  a screen is explicitly T2 (SQL Console, audited reads). See root `CLAUDE.md`.

## How to open the prototype

```bash
open docs/design/Lynceus.dc.html   # macOS; support.js loads from the same dir
```

Toggle prototype props (in the HTML) to exercise states before comparing:
`fleetState` (`unhealthy`|`healthy`), `enablePostgres`/`enableRedis`/`enableValkey`/
`enableElasticsearch`/`enableOpensearch`, `accentColor`, `defaultTheme`.

### Per-area gaps (verified)

#### Design system & tokens — impl: **partial**, parity: **none**

Gaps:
- No CSS design-token layer: every color/spacing value is an ad-hoc hardcoded hex, not a custom-property token set matching the README dark/light palettes (bg #0c1118, surface #10161f, line #26303d, text #dbe4ee, acc #2dd4bf, crit/warn/info pairs, etc.)
- Fonts wrong and absent: Work Sans (UI 13px) and JetBrains Mono (data/labels 9.5–10.5px letter-spaced, tabular-nums) are neither loaded nor applied; no font-role split between prose and data
- No dark theme at all — implementation is light-only; design requires dark as the DEFAULT plus a light variant
- No theme mechanism: no data-theme attribute, no @media prefers-color-scheme, no 'system' initial preference, no toggle wiring
- No accent presets: Teal/Cyan/Indigo with per-theme bright(dark)/deep(light) variants and matching bg/hover tints are entirely missing
- Shape language non-conforming: 8/6/9px radii instead of 2px (1px tiny badges); no unrounded 8px severity squares; no 24px icon buttons; shadow policy (none except dropdowns/modals 0 8px 24px) not encoded
- Fonts/token host not self-hosted for the privacy-first stance (htmx already pulled from unpkg CDN at layout.templ:25) — a design-token/font strategy must avoid external hosts

#### Global top bar — impl: **absent**, parity: **none**

Gaps:
- No 48px top-bar shell exists at all; the shell is an h1 title plus a flat horizontal nav
- Design tokens absent/wrong: light theme + system-ui font instead of dark-default (with light/system) JetBrains Mono
- No logo mark (L monogram + LYNCEUS wordmark) that returns to Fleet
- No SCOPE picker button (SCOPE: <label> chevron) and no searchable dropdown of scopeable entities with kind badges
- No search matching across names + kinds + provider (aws/azure)/engine metadata
- No back-to-FLEET reset affordance and no accent-color scoped state on the chip
- No time-range segmented control (15M/1H/24H/7D/30D) wired to a shared range param
- No visible poll indicator (POLL 3S / UPDATED Ns AGO with pulsing dot)
- No theme toggle (dark default / light / system initial preference) with persistence
- No user button/menu: identity header (username + GROUP + T2 status), GOVERNANCE (Audit Log, Access & Roles), ADMIN (Provider Setup, Collectors, Data & Retention, Settings), Sign out

#### Sidebar + scope-driven nav rebuild — impl: **partial**, parity: **none**

Gaps:
- No scope-driven sidebar engine: nav does not rebuild by scope level; layout.templ ships one flat static list to every page
- Fleet-scope nav tree absent: no OVERVIEW(Fleet) + per-vertical DATABASE/SEARCH/CACHE sections + CONSOLE(Saved Scripts only); low-level sections are not suppressed at fleet because there is no fleet scope at all
- Cluster-scope tree is a stub: ClusterSidebar has 5 flat links vs the design's grouped tree (CLUSTER:<name> {Overview,Nodes,Databases,Capabilities} · QUERIES · ADVISORS{Index,Vacuum,Config·per node} · ACTIVITY · CONSOLE{SQL Console+Saved Scripts} · CHECKS & ALERTS · SCHEMA · LOGS)
- Node, Pooler, and Database scope nav trees do not exist (no NODE:/POOLER:/DATABASE: groups, no scoped page shells or routes)
- Nav gating rules unimplemented: Saved Scripts is not present at any scope; SQL Console entry does not exist; no rule enforcing low-level sections hidden at fleet
- Sidebar design-system styling absent: no 208px JetBrains Mono rail, no faint letter-spaced section headers, no SOON/T2 badges, no accent-text + tinted-bg + 2px right-border active item
- Per-engine section visibility (SEARCH shows if ES||OS, CACHE if Redis||Valkey) has no nav wiring

#### Scope model core — impl: **partial**, parity: **none**

Gaps:
- No 'current scope' concept exists anywhere (no server- or client-side scope state, no scope resolver, no scoped URL scheme)
- No top-bar app shell: missing logo→fleet, searchable SCOPE picker dropdown (clusters/nodes/poolers/cluster-qualified databases with kind badges, search over name+kind+provider+engine), accent scope chip, ← FLEET button, time-range control, poll indicator, theme toggle, user menu
- Nav is not rebuilt per scope level — a single flat top-nav shows all low-level sections (advisors/checks/waits) at all times, directly contradicting 'low-level sections never appear at fleet scope'
- No node scope, pooler scope, or database scope at all — only a two-level fleet-list → cluster-detail structure exists; database identity is not modeled as cluster+name in routing
- No Fleet Overview surface — '/' is top-queries; there is no fleet triage screen for scope-set deep links to land beside
- Wrong scope-set interaction: rows/cards are clickable links instead of carrying a ⌖ icon button on otherwise-inert rows
- Scoped Overview 'OPEN ISSUES ON THIS <CLUSTER|NODE|DATABASE>' list is absent
- Deep-link-to-explanation only exists as an in-cluster HTMX query drilldown (web/overview.templ:183-208); there is no fleet/needs-attention item that sets scope AND opens the specific check/insight/advisor explanation
- Design tokens/shell styling absent — even the cluster views that exist ignore the design system, so designParity is none, not merely partial

## Tracking plan (beads)

The backend/data for most areas is already tracked under the M2–M6 epics; the **UI/design
layer** is what the handoff adds and is otherwise untracked. Intended structure — one epic
+ coarse workstream children, each carrying `needs-plan` until a plan lands under
`docs/superpowers/plans/`:

**Epic:** `[epic] UI design parity — implement the design handoff (shell, scope, verticals)` (P1)

| # | Child (workstream) | P | Depends on |
|---|--------------------|---|------------|
| F1 | Design-token CSS layer + self-hosted fonts + theme mechanism (dark-default/light/system) | 1 | — |
| F2 | Global top bar + scope model + searchable SCOPE picker | 1 | F1 |
| F3 | Scope-driven sidebar nav rebuild (per-scope nav trees) | 1 | F2 |
| D  | Fleet dashboard — triage strips, needs-attention, problem-only cards, healthy/unhealthy | 1 | F2, F3 |
