# ly-xqf.14 — auto_explain plan extraction Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: use superpowers:subagent-driven-development or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Turn the `auto_explain.plan` log events already produced by the log-parsing framework (`ly-cxe.1`, `EventAutoExplainPlan` + `LogPayload`) into **structured, normalized (T1) query-plan records** — a recursive node tree of plan-node types, relation/index names, estimated and (when `ANALYZE`) actual costs/rows/times — that **physically cannot carry a query literal**. This is the foundation every M3 EXPLAIN insight reads (it blocks 13 downstream issues: `ly-u4t.1–.11`, `ly-7ck.13/.15`).

**Architecture:** auto_explain writes the chosen query's plan into the Postgres log. The log framework already frames + classifies these lines as `EventAutoExplainPlan` and routes the plan body into the (T2, collector-local) `LogPayload`. This feature adds a **plan extractor** that consumes the payload's plan body, parses it (JSON format — see decision below), and emits a normalized `QueryPlan` proto. Literal-bearing strings in the plan (`Filter`, `Index Cond`, `Hash Cond`, `Join Filter`, `Recheck Cond`, `Output`) are run through the existing `normalize` package (or dropped) so no literal survives into T1. A new range-partitioned `query_plans` table lands the data. The collector wires the extractor into the log pipeline; the ingestion server persists the `QueryPlan` messages carried on the `Snapshot`.

**Decision — require JSON plan format.** auto_explain can emit `text|json|xml|yaml`. We support **JSON only** (`auto_explain.log_format = json`) for extraction:
- JSON is a structured tree → no fragile text-indent parsing; node fields are explicit (`Node Type`, `Relation Name`, `Plan Rows`, `Total Cost`, `Actual Total Time`, `Plans[]`).
- Text format is ambiguous to parse robustly and is exactly the kind of 20-lines-to-do-badly that the goal warns against.
- If a non-JSON `auto_explain.plan` payload arrives, the extractor returns `(nil, ErrUnsupportedPlanFormat)` and the collector logs + drops it (no partial/guessed data). The capability-discovery feature (`ly-xnk`) will surface a recommendation to set `log_format = json`.

**Privacy (defense in depth):**
- **Source layer:** the extractor never copies `Filter`/`Output`/cond strings verbatim into T1 — each is normalized (literals → `$n`) via `normalize.Normalize`; if normalization fails (`Tier != TierNormalized`), the field is **omitted**, never shipped raw.
- **Wire layer:** the `PlanNode`/`QueryPlan` proto field allowlist is enforced by a new contract test (`TestQueryPlanHasNoLiteralFields`, `TestPlanNodeConditionsAreNormalized`) — mirrors the existing `TestQueryStatHasOnlyNormalizedFields`.
- **Storage layer:** `query_plans` stores the normalized tree as JSONB plus scalar columns; no raw-plan-text column exists. The original literal-bearing plan body stays in the collector-local `LogPayload` (T2) and is only shipped if the separate T2 path is explicitly enabled + audited.

**Tech Stack:** Go 1.26, `encoding/json` (stdlib — plan tree is JSON), `pganalyze/pg_query_go/v6` via the existing `internal/normalize`, `protoc-gen-go`, vanilla PostgreSQL 16 range partitioning, `testcontainers-go` for the store integration test. The plan parser itself is pure Go and unit-tested without a DB (parse fixtures captured from real auto_explain JSON output).

**Specs:**
- `docs/specs/2026-05-29-lynceus-design.md` §2 (privacy backbone), §3 (collector), §3.5 (vanilla-Postgres stats DB).
- `docs/specs/2026-05-29-lynceus-features.md` — EXPLAIN insights (M3) read these records.

### Storage decision (justification)

A **new partitioned table `query_plans`**, not a column on `query_stats`:
- Different grain: a plan is per-execution-sample (captured when auto_explain fires), not per-fingerprint-aggregate. Forcing it into `query_stats` would break that table's `(server_id, fingerprint)` aggregate grain.
- The normalized node tree is naturally a document → store as `JSONB` (queryable by the M3 insight passes via `->>`/`@>` without a join explosion), plus a few hot scalar columns (`fingerprint`, `total_cost`, `actual_total_time_ms`, `captured_at`) for indexed filtering.
- Same weekly range-partition strategy as `query_stats`/`activity_buckets` → reuse the established `isoWeekBounds`/partition-name pattern. BRIN on `captured_at`, btree on `(server_id, fingerprint, captured_at)`. RDS/Aurora-safe, no extensions.

```proto
syntax = "proto3";
package lynceus.v1;
option go_package = "github.com/dobbo-ca/lynceus/internal/proto/lynceus/v1;lynceusv1";

// QueryPlan is one extracted auto_explain plan, normalized at the collector.
// INVARIANT: no field may carry a query literal. Condition strings are
// normalized (literals -> $n) or omitted. See contract_test.go.
message QueryPlan {
  string fingerprint            = 1; // fingerprint of the normalized statement the plan is for
  int64  captured_at_unix       = 2;
  int32  format_version         = 3; // plan extractor schema version (start at 1)
  double total_cost             = 4; // root Total Cost
  double actual_total_time_ms   = 5; // root Actual Total Time (0 if not ANALYZE)
  PlanNode root                 = 6;
}

// PlanNode is one node in the plan tree. Recursive via `plans`.
message PlanNode {
  string node_type               = 1;  // "Seq Scan", "Index Scan", "Hash Join", ...
  string relation_name           = 2;  // table, if any
  string index_name              = 3;  // index, if any
  string alias                   = 4;  // range-table alias (identifier, not a literal)
  string join_type               = 5;  // "Inner" | "Left" | ... ; "" if not a join
  string scan_direction          = 6;  // "Forward" | "Backward" | ""

  double startup_cost            = 7;
  double total_cost              = 8;
  int64  plan_rows               = 9;
  int32  plan_width              = 10;

  double actual_startup_time_ms  = 11; // 0 unless ANALYZE
  double actual_total_time_ms    = 12;
  int64  actual_rows             = 13;
  int64  actual_loops            = 14;

  // Filter/Index Cond/Hash Cond/etc. AFTER normalization (literals -> $n).
  // Empty if the source condition could not be safely normalized.
  string normalized_condition    = 15;

  repeated PlanNode plans        = 16;
}
```

## Task 2: Condition normalizer (reuse internal/normalize)

**Files:** create `internal/planextract/normalize_cond.go` + tests in `extract_test.go`.

auto_explain condition strings look like `(orders.status = 'shipped'::text)` or `(id = ANY ('{1,2,3}'::integer[]))`. These are boolean expressions, not full statements, so wrap them before handing to `normalize.Normalize`.

- [ ] **Step 1: failing unit test:**

```go
func TestNormalizeCondition_stripsLiterals(t *testing.T) {
	cases := []struct{ in, want string }{
		{"(status = 'shipped'::text)", "(status = $1)"},          // exact form may differ; assert no literal remains
		{"(id = ANY ('{1,2,3}'::integer[]))", ""},                // arrays may be un-normalizable -> omit
	}
	for _, c := range cases {
		got := planextract.NormalizeCondition(c.in)
		if strings.ContainsAny(got, "'") || strings.Contains(got, "shipped") || strings.Contains(got, "{1,2,3}") {
			t.Errorf("literal leaked: %q -> %q", c.in, got)
		}
	}
}
```

The hard assertion is **"no literal survives"** (a `'`-quoted string or numeric array never appears in output). Exact placeholder form is secondary — the test asserts the safety property, not a golden string.

- [ ] **Step 2: implement `NormalizeCondition(cond string) string`:** wrap as `SELECT 1 WHERE <cond>` (or `SELECT CASE WHEN <cond> ...`), run `normalize.Normalize`, and if `Tier == TierNormalized` extract the normalized predicate back out; otherwise return `""`. Prefer returning `""` over any risk of a leak — **fail closed**.
- [ ] **Step 3:** tests PASS. Commit `feat(planextract): fail-closed condition normalizer`.

## Self-Review

**1. Spec coverage** — privacy enforced at source (fail-closed condition normalizer), wire (allowlist contract test), and storage (no raw-plan column). Vanilla-Postgres partitioned table, RDS-safe. JSON-only decision documented + surfaced via capability discovery.

**2. Hooks for adjacent features** — every M3 EXPLAIN insight (`ly-u4t.*`) reads `query_plans.plan_tree` (JSONB) + scalars; no new schema needed for them. `TopPlansByQuery` is their entry point. Plan comparison (`ly-xqf.11`) diffs two `plan_tree` documents.

**3. Out of scope (confirmed)** — no insight logic (separate M3 beads); no plan *visualization* (that is `ly-xqf.10`); no T2 raw-plan shipping path; no text/xml/yaml plan formats.

**4. Privacy fail-closed check** — `NormalizeCondition` returns `""` on any normalization failure; `Extract` never copies `Output`/`Query Text`; the contract test makes adding a raw field a build failure. The literal-bearing source plan stays in the collector-local T2 `LogPayload`.

**5. Consistency** — writer uses `CopyFrom` (matches `ly-3na`); partition naming reuses `isoWeekBounds` with a `query_plans_` prefix distinct from `query_stats_` / `activity_buckets_`; proto field names match the contract-test allowlist exactly.
